## 1. Summary

Install/upgrade for v2.2.0 is **mostly a mechanical extension** of the existing v2.1.2 scripts. Schema changes are already additively covered on every live install we have data for (pi02 and pi03 both already carry the new `cad_events.cad_hw_*` / `cad_sw_*` and `packets.lbt_*` columns). The main actual work required in the scripts is:

1. **Add two new overlay files** to the copy loops in both `install.sh` and `upgrade.sh`:
   - `overlay/pymc_repeater/repeater/metrics_retention.py`
   - `overlay/pymc_repeater/repeater/web/packet_trace.py`
2. **Resolve a config-key direction conflict** between `config/config.yaml.template` (which uses `bridge.dedup_ttl_seconds`) and the migration logic in `upgrade.sh` (which renames `dedup_ttl_seconds` back to `dedup_ttl`). One of these is wrong.
3. **Pre-create `/tmp/pymc_spectral_results.json`** with `pi:pi` ownership (already done at `upgrade.sh` lines 618–622 — verify for install.sh).
4. **Add light extra safeguards**: DB backup before migration, explicit version-downgrade refusal, post-upgrade journal health check, explicit ALTER TABLE fallback for `cad_events.cad_hw_*` / `cad_sw_*` (defensive for very old installs).

No HAL changes in the build process beyond what's already in v2.1.2 (HAL 2.10, fork-based). The HAL overlay C-file list is unchanged and already covers the files modified this cycle.

## 5. Config Changes

### 5a. Template diff vs v2.1.2 (`config/config.yaml.template`)

```diff
 bridge:
   bridge_rules: []
-  dedup_ttl: 15
+  dedup_ttl_seconds: 300

 repeater:
   advert_interval_seconds: 133200
   allow_discovery: true
-  cache_ttl: 30
+  cache_ttl: 60
+  max_cache_size: 1000
```

### 5b. 🔴 Direction conflict to resolve BEFORE release

The template defines the new canonical name as **`bridge.dedup_ttl_seconds`**, but the migration code in `upgrade.sh` (lines 787–798) does the opposite — it pops `dedup_ttl_seconds` and writes back `dedup_ttl: 15`:

```python
# upgrade.sh — current migration (INVERTED for v2.2.0)
if 'dedup_ttl_seconds' in br:
    old_val = br.pop('dedup_ttl_seconds')
    if 'dedup_ttl' not in br:
        br['dedup_ttl'] = 15
    changes.append('bridge.dedup_ttl_seconds->dedup_ttl')
```

This is consistent only if the code still reads `bridge.dedup_ttl`. Live pi02 config actually contains **both**:

```yaml
bridge:
  dedup_ttl: 15
  dedup_ttl_seconds: 300
```

**Recommended resolution** (agent suggestion, not committed):

- Decide canonical key name. The SSOT story in the release notes names **`bridge.dedup_ttl_seconds`** as the new canonical key.
- Flip the migration to: if `dedup_ttl` present and `dedup_ttl_seconds` missing → copy value into `dedup_ttl_seconds` (seconds numeric), then delete `dedup_ttl`.
- Audit runtime readers (`repeater/config.py`, `engine.py`, `wm1303_api.py`) and align them on `dedup_ttl_seconds`.
- Until that code audit is done, the safe short-term fix is to **keep both keys** and have code read `dedup_ttl_seconds` with `dedup_ttl` as fallback. That is the minimum work needed to ship v2.2.0 without regressions.

### 5c. New keys to be merged into existing `/etc/pymc_repeater/config.yaml`

All handled by the deep-merge logic at `upgrade.sh` lines 870–914 (`deep_merge(tmpl, live)`). No new logic needed, **provided** §5b is resolved.

| Key | Default | Notes |
|---|---|---|
| `bridge.dedup_ttl_seconds` | `300` | See §5b conflict. |
| `repeater.cache_ttl` | `60` (template) | Previously `30`; migration currently clamps any value ≥ 60 down to 30. That's also misaligned and should be re-examined. |
| `repeater.max_cache_size` | `1000` | New key, deep-merge adds it safely. |

### 5d. Optional key (not in template, safe to omit)

## 6. DB Schema Changes

### 6a. Baseline vs current — finding

Both pi03 (v2.1.2) and pi02 (latest) already carry an **identical schema** for:

- `cad_events` — columns `cad_clear, cad_detected, cad_skipped, cad_hw_clear, cad_hw_detected, cad_sw_clear, cad_sw_detected`
- `packets` — columns `lbt_attempts, lbt_backoff_delays_ms, lbt_channel_busy`

Meaning the v2.2.0 schema matches v2.1.2 as deployed. No user-visible schema delta.

### 6b. Fresh install path (v2.2.0)

Covered by `CREATE TABLE IF NOT EXISTS cad_events (...)` at `upgrade.sh` lines 1159–1170 which lists all 7 columns. ✅

### 6c. Defensive ALTER TABLE (recommended addition)

For very old installs (pre-v2.1.0) whose `cad_events` table predates the HW/SW split, the current migration code will **not** add the missing columns because it only does `CREATE TABLE IF NOT EXISTS`. Add defensive `ALTER TABLE ... ADD COLUMN` entries to `col_migrations` list at `upgrade.sh` lines 1189–1196:

```python
col_migrations = [
    ('adverts', 'zero_hop', 'BOOLEAN NOT NULL DEFAULT FALSE'),
    ('packets', 'lbt_attempts', 'INTEGER DEFAULT 0'),
    ('packets', 'lbt_backoff_delays_ms', 'TEXT'),
    ('packets', 'lbt_channel_busy', 'BOOLEAN DEFAULT FALSE'),
    ('channel_stats_history', 'noise_floor_dbm', 'REAL'),
    ('channel_stats_history', 'pkt_count', 'INTEGER'),
    # NEW in v2.2.0 — defensive for pre-v2.1 installs
    ('cad_events', 'cad_hw_clear',     'INTEGER DEFAULT 0'),
    ('cad_events', 'cad_hw_detected',  'INTEGER DEFAULT 0'),
    ('cad_events', 'cad_sw_clear',     'INTEGER DEFAULT 0'),
    ('cad_events', 'cad_sw_detected',  'INTEGER DEFAULT 0'),
]
```

`has_column()` already guards against double-adds → idempotent.

### 6d. VACUUM recommendation

The first run of `metrics_retention` on an upgraded install with the previous 31-day retention may delete a meaningful row count. `metrics_retention.py` runs its own weekly VACUUM, so no script-side VACUUM is strictly needed. If the user wants a one-time cleanup immediately after upgrade (nicer in disk-pressure environments), add:

```bash
# upgrade.sh — end of Phase 7b, only for the first upgrade to v2.2.0
if [ "${CURRENT_VERSION}" != "v2.2.0" ] && [ "${UPGRADE_VERSION}" = "v2.2.0" ]; then
    step "One-time VACUUM after retention cutover"
    ${VENV_DIR}/bin/python3 -c "import sqlite3; sqlite3.connect('${DB_PATH}').execute('VACUUM')" >>"${LOG_FILE}" 2>&1 || true
    ok "VACUUM complete"
fi
```

### 6e. Idempotency

All schema mutations use `IF NOT EXISTS` + `has_column()` guards → running the migration twice is safe. ✅

## 8. Service Lifecycle

Existing sequence (already sound):

1. Pre-upgrade backup of `CONFIG_DIR` and `PKTFWD_DIR` to `/home/pi/backups/pre-upgrade-<ts>/`.
2. `systemctl stop pymc-repeater`.
3. System prerequisites + rrdtool venv symlink + NTP + I2C checks.
4. Repo pulls → HAL overlay → Python overlays → HAL rebuild (conditional) → pip reinstall.
5. pycache sweep + `/tmp/pymc_*.json` pre-create.
6. Config merge/migrate.
7. DB schema migration + data sanity cleanup.
8. `systemctl start pymc-repeater`, wait 3 s, check `is-active`, curl web port.

### Gaps (recommended extra safeguards)

| # | Safeguard | Where to add |
|---|---|---|
| 1 | **DB backup** before Phase 7b | After `Phase 2` in `upgrade.sh`, `cp /var/lib/pymc_repeater/*.db ${UPGRADE_BACKUP}/db/` |
| 2 | **Version-downgrade refusal** | After current-version detection (line ~150), compare `UPGRADE_VERSION` vs `CURRENT_VERSION` with a simple `sort -V` and refuse if upgrade is a downgrade |
| 3 | **Post-upgrade journal scan** | After `is-active` check (line ~1339), `journalctl -u pymc-repeater --since "30 seconds ago" -p err --no-pager` and warn if errors found |
| 4 | **Health check for metrics_retention** | After start, optionally `journalctl -u pymc-repeater | grep -m1 "MetricsRetention started"` with short timeout |
| 5 | **Rollback hint in log** | On `fail()`, print location of `UPGRADE_BACKUP` and one-line rollback command |

## 12. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| User on heavily customized config (e.g. renamed keys, manual edits) | Medium | Deep-merge only **adds** missing keys; existing user values preserved. Dedup-TTL direction conflict is the only real risk — resolve per §5b before release. |
| User with corrupted DB | Low | Add DB backup before Phase 7b (§8 gap 1). Migration code uses transactions and commits only on success. |
| User on very old version (pre-v2.0) with non-additive schema drift | Low (no known reports) | Defensive ALTER TABLE for `cad_events` HW/SW columns (§6c) covers the one known long-range gap. |
| First-boot `/tmp/pymc_spectral_results.json` ownership mismatch | Low | Pre-create step at `upgrade.sh` 618–622 exists. Mirror into `install.sh` (§9 item 3). |
| HAL rebuild failure leaves device with no `lora_pkt_fwd` | Very Low | `BINARY_MISSING` branch forces rebuild; `FORCE_REBUILD` available; pre-upgrade backup preserves the previous binary. |
| Retention cutover deletes too much on hand-tweaked installs | Low | `metrics_retention` logs per-table row counts; one-time VACUUM optional. User can override `storage.retention.metrics_days` if needed. |
| User upgrading a device that hasn't been touched in months (many stale rows) | Medium | First retention pass takes longer but runs in a daemon thread with `timeout=10` per table — will not block boot. |
